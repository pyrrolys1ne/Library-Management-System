## 一、项目背景与目标

### 1.1 背景
高校图书馆在日常运营中需要管理大量图书资源、学生借阅、预约排队、逾期罚款等业务。传统手工管理方式效率低下，容易产生遗漏与错误。本项目旨在构建一套基于 Web 的图书馆信息管理系统，通过数据库事务、触发器、存储过程等机制保证数据一致性与业务流程自动化。

### 1.2 目标
- 为学生提供便捷的图书检索、借阅、预约、历史查看服务
- 为管理员提供还书核销、罚款处理、图书底库管理等功能
- 通过数据库编程（存储过程、触发器、函数）实现核心业务逻辑的 ACID 保障
- 前端采用现代化响应式框架，提供良好的用户体验

## 五、业务规则

### 5.1 借书规则
1. 学生账户状态必须为 **Normal**，被冻结 (Suspended) 的学生不可借书
2. 学生必须无未缴清的超期罚款 (PenaltyInfo.Pstatus = 'Unpaid')
3. 图书在馆库存 Bcount > 0 方可借出
4. 借阅期限为 30 天 (Due_date = Borrow_date + 30)
5. 以上校验在存储过程 `sp_borrow_book` 中以单事务 + 行级锁 (`FOR UPDATE`) 执行，防止并发超借

### 5.2 还书与逾期处理规则
1. 还书时更新 BorrowRecord.Return_date，由触发器 `trg_after_return` 自动核算
2. 若 Return_date > Due_date，超期天数 = DATEDIFF(Return_date, Due_date)
3. 罚款金额 = 超期天数 × ¥0.50
4. 超期自动生成 PenaltyInfo 记录并将学生 Sstatus 设为 Suspended
5. 还书后检查该书的预约队列：
   - 若有有效预约 (Rstatus='Active' 且未过期)，自动转借给首位预约者，库存不变
   - 若无预约，库存 Bcount + 1

### 5.3 预约规则
1. 仅当图书库存 Bcount = 0 时可发起预约
2. 同一学生对同一本书不可重复预约 (Sno + Bno + Active 唯一)
3. 预约有效期 30 天 (Expire_date = Reserve_date + 30)
4. 入库触发预约消费：库存从 0 增为正数时，自动将首位有效预约转为借出状态
5. 被冻结学生不可预约

### 5.4 罚款与解冻规则
1. 罚单按笔管理：每笔对应一次借阅流水号
2. 管理员核销罚单后标记 Pstatus = 'Paid'
3. 核销后检查该学生是否无剩余 Unpaid 罚单，若无则自动解冻 (Sstatus → Normal)
4. 解冻后学生恢复借书与预约权限

### 5.5 图书管理规则
1. 新书入库时书号自动递增分配 (B001 → B002 → ...)
2. 库存增减可通过管理员手动操作
3. 当库存与在借量均为 0 时，可彻底删除该书目
4. 管理员入库时若有人排队，自动触发预约流转
5. 上传的封面按 `{书号}_cover.{ext}` 归档，PDF 按 `{书号}_preview.{ext}` 归档

## 六、非功能需求

| 编号 | 类别         | 描述 |
| ---- | ------------ | ---- |
| N01  | 数据一致性   | 借书操作使用数据库事务 + 行级锁，保证并发场景下库存不被超借 |
| N02  | 自动化       | 超期罚款、预约流转、账号冻结/解冻均通过触发器和存储过程自动完成 |
| N03  | 响应式界面   | 基于 Element Plus 组件库，支持主流浏览器 |
| N04  | 连接池       | 使用 DBUtils PooledDB，最大 20 连接，支持 Flask 多线程并发 |
| N05  | 文件管理     | 上传文件按书号规范化命名归档，便于后续维护 |
| N06  | 安全         | Session 会话管理，密码字段独立存储（生产环境建议加密） |

## 七、系统边界

### 本项目包含
- ✅ 学生端：馆藏查询、图书详情、借书、预约、借阅历史
- ✅ 管理端：还书核销、罚款处理、图书底库 CRUD、搜索
- ✅ 数据库编程：存储过程、触发器、函数
- ✅ 封面上传与电子版归档

### 本项目不包含
- ❌ 用户注册（学生/管理员由后台直接录入）
- ❌ 邮件/短信通知（预约到书提醒等）
- ❌ 图书续借功能
- ❌ 评论/评分系统
- ❌ 细粒度权限控制（目前仅区分 student / admin 两种角色）
- ❌ 生产级安全防护（密码明文存库）
